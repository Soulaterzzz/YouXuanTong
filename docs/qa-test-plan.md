# 优选通保险服务平台测试方案

## 1. 测试目标

- 基于 `README.md` 中定义的 Spring Boot + Vue 架构，建立“接口层 + 前端逻辑层 + 并发单元层”三层测试体系。
- 覆盖核心业务的功能、边界、异常、权限和并发风险，优先保障登录鉴权、管理员操作、投保激活、通知发布等高风险链路。
- 输出可直接执行的自动化代码，并给出可落地的手工测试用例与风险清单。

## 2. 测试范围

### 2.1 后端范围

- 认证接口：登录、退出、当前用户。
- 用户端接口：统计、产品激活、费用/保险/充值查询。
- 管理端接口：用户管理、产品管理、批量生效、通知发布。
- 公共接口：通知公告列表。

### 2.2 前端范围

- 路由鉴权守卫。
- 管理端页面中批量生效的前端校验逻辑。
- 管理端状态展示逻辑。

### 2.3 非本轮覆盖范围

- MySQL 真库联调与大数据量性能压测。
- 浏览器兼容性、弱网、文件上传链路的 E2E 自动化。
- 报表导出文件内容比对。

## 4. 环境与数据

### 4.1 环境建议

- JDK：17+
- Maven：3.6+
- Node.js：18+
- MySQL：8.0+（用于业务联调，自动化单测不强依赖）

### 4.2 账号与数据

- 管理员：`admin / admin111`
- 普通用户：`user1 / user123`
- 建议准备以下专项数据：
  - 1 个已启用管理员账号
  - 2 个普通用户（1 个余额充足、1 个余额不足）
  - 3 个产品（在售、下架、库存边界）
  - 3 张保单（待审核、待生效、已生效）
  - 2 条通知（短标题、长内容）

## 8. 并发与稳定性测试

### 8.1 已自动化并发场景

| 场景 | 目标 | 自动化 |
|------|------|--------|
| token 并发签发 | 验证 `ConcurrentHashMap` 线程安全和 token 唯一性 | `AuthTokenServiceTest` |
| token 撤销 | 验证登出后 token 失效 | `AuthTokenServiceTest` |

### 8.2 建议压测/并发场景

| 场景 | 风险 | 建议 |
|------|------|------|
| 同一用户并发激活产品 | 余额被重复扣减、库存超卖 | 引入事务隔离、悲观锁或余额版本号 |
| 同一保单重复批量生效 | 保单号重复使用、状态穿透 | 后端增加状态机校验与幂等 token |
| 多管理员并发发布通知 | 最终显示顺序和内容被覆盖 | 增加更新时间版本控制 |
| 并发上下架产品 | 用户端读取到脏状态 | 上下架接口增加状态校验与更新时间戳 |

## 9. 风险分析

| 风险项 | 等级 | 现象 | 影响 | 建议 |
|--------|------|------|------|------|
| `Admin.vue` 旧逻辑与后端路径不一致 | 高 | 调用 `/admin/*` 而非 `/api/admin/*` | 管理端页面请求失败 | 已在本轮修正为 `/api/admin/*` |
| 管理端批量生效仅前端校验日期先后 | 高 | 绕过前端可直接提交逆序日期 | 产生脏数据 | 后端补充日期规则校验 |
| 充值/激活部分 DTO 缺少 Bean Validation | 高 | 空值和非法金额可能进入业务层 | 数据质量与异常处理风险 | 为 DTO 增加 `@NotNull/@DecimalMin` |
| 登录 token 存储在内存 | 中 | 服务重启后 token 全失效、无法横向扩展 | 会话不稳定 | 迁移 Redis 或集中式 session |
| 现有自动化以接口层为主 | 中 | 未覆盖数据库事务级联 | 缺陷可能延后暴露 | 下一阶段接入 Testcontainers |
