# 优选通保险分销平台后续开发文档

> 目标：基于当前仓库现状，整理可持续开发所需的工程规范、目录约定、配置策略、数据库演进方式和测试要求，作为后续新增功能和联调排障的统一参考。

## 1. 项目现状概览

当前仓库是一个典型的前后端分离保险业务平台，后续目标是演进为“保险分销平台”。主要特征如下：

- 后端：`Spring Boot 3.5 + MyBatis-Plus + MySQL`
- 前端：`Vue 3 + Vite + Element Plus`
- 认证方式：`Session / Cookie`，不是 JWT
- 数据库演进：既有初始化脚本，也有增量迁移脚本和启动时兼容修复逻辑
- 质量保障：后端已有接口测试与 token 并发测试，前端已有路由守卫测试

当前已有的文档位于 `docs/` 目录，说明项目已经具备一定的交付和测试意识，但缺少一份专门面向“后续开发”的工程规范文档，因此本文件补齐这一块。

## 分销平台改造目标

这一轮改造不是简单改名，而是把现有平台从“面向终端用户的保险服务平台”调整为“面向分销商的保险分销平台”。

### 角色定义

- 原有 `用户` 角色统一迁移为 `分销商`
- `管理员` 角色保留，负责商品池、账号、审核、统计和全局配置
- 前端菜单、页面文案、接口说明、测试用例里的“用户”应逐步替换为“分销商”

### 产品能力目标

- 分销商可以在自己可见的产品范围内独立修改产品名
- 分销商可以独立设置销售价格
- 产品名称和价格修改后，只影响后续销售与展示，不应回写历史保单与历史费用
- 历史订单、保险记录、费用记录必须保存“下单时快照”，避免后续改价或改名污染历史数据

### 导出能力目标

- 后台支持批量导出费用清单为 Excel
- 后台支持导出单个保险清单为 PDF
- 导出操作应支持基于筛选条件、状态和时间范围的精确输出
- 导出文件名应包含业务类型和时间信息，便于下载和留档

### 当前代码的直接影响点

- `AnXinXuanService.exportInsurances` 目前是空实现，需要补齐
- 现有的费用清单/保险清单列表接口要复用查询条件，避免导出和列表口径不一致
- 现有产品与保单字段需要补充“分销商视角”的价格和名称快照
- 现有前端页面中“用户管理”“用户端”等文案要同步改造为“分销商管理”“分销商端”

### 推荐的数据建模方式

- 平台商品池保留为基础产品数据
- 分销商侧新增产品配置层，保存分销商自定义的产品名、销售价、状态和授权范围
- 订单、保单、费用记录在生成时写入快照字段，后续展示统一读取快照
- 如果暂时不适合重构整套表结构，至少先补“分销商名称/价格覆盖层”，再逐步做表结构收敛

## 3. 工程分层规范

### 3.1 后端分层职责

- `controller` 只负责接收参数、调用 service、返回统一响应，不写复杂业务。
- `service` 负责业务编排，包含校验后的核心流程。
- `service/impl` 负责具体实现，优先保持一个 service 对应一个业务域。
- `mapper` 只做持久化访问，不承载业务判断。
- `entity` 只映射表结构，不混入页面展示逻辑。
- `dto` 专门承载请求入参、批量操作参数和校验注解。
- `vo` 专门承载接口返回结构，避免直接暴露 entity。
- `query` 专门承载列表查询、分页查询、筛选条件。

### 3.2 前端分层职责

- `views` 放页面级组件。
- `router` 放路由表与守卫。
- `composables` 放可复用的业务逻辑。
- `styles` 放全局样式和公共样式变量。
- 页面内部的通用部件建议继续拆分成局部 `components`，避免视图文件过大。

## 4. 命名规范

### 4.1 Java 命名

- 实体类统一使用 `*Entity`
- 返回对象统一使用 `*VO`
- 请求对象统一使用 `*Request`
- 查询对象统一使用 `*Query`
- 控制器统一使用 `*Controller`
- 服务接口统一使用 `*Service`
- 服务实现统一使用 `*ServiceImpl`
- Mapper 统一使用 `*Mapper`

### 4.2 接口命名

- 所有后端接口统一使用 `/api` 前缀。
- 用户端、管理员端、公共接口要区分清楚，避免前端拼接路径不一致。
- 新增接口优先使用清晰、可读、可维护的动词式路径，例如：
  - `POST /api/admin/notices/publish`
  - `PUT /api/admin/users/{userId}/enable`
  - `POST /api/anxinxuan/products/activate`

### 4.3 前端命名

- 页面文件名要能体现业务含义，例如 `Home.vue`、`Login.vue`
- 路由名、菜单名、页面标题应保持一致
- 组件名尽量与页面语义一致，不要用泛化命名

## 5. 编码与接口规范

### 5.1 返回值规范

当前项目统一使用 `ApiResponse<T>` 包装接口返回值，后续开发也应保持一致：

- 成功返回统一格式
- 业务错误返回统一错误码
- 不要在控制器中直接返回裸对象或裸字符串

### 5.2 参数校验规范

- 所有外部入参优先加 `jakarta.validation` 校验注解
- 必填字段、长度限制、数值边界、日期先后关系都应在入参层拦截
- 对于复杂规则，建议在 service 层补充二次校验

### 5.3 异常处理规范

- 业务异常使用 `BusinessException`
- 参数异常由全局异常处理器统一转换为标准响应
- 非预期异常由全局异常处理器兜底，不允许直接抛到前端

### 5.4 鉴权规范

- 当前系统采用 Session / Cookie + 令牌上下文，不要随意切换为 JWT
- 需要登录态的接口必须通过统一鉴权上下文获取当前用户
- 管理端接口必须校验管理员权限，前端路由守卫只作为第一道防线，不能替代后端鉴权

## 6. 数据库与迁移规范

### 6.1 推荐策略

- `init_db.sql` 只用于初始化全量环境
- `src/main/resources/sql/` 中的脚本用于增量演进
- 需要兼容老库时，优先增加迁移脚本，不要直接依赖手工改表

### 6.2 现有约定

- 启动时会执行数据库兼容修复逻辑，补齐必要表结构和索引
- 逻辑删除使用 `deleted` 字段
- 数据库字段名与 Java 属性名之间统一开启下划线转驼峰映射

### 6.3 新增表或字段时的要求

- 先确认是否可以通过现有表扩展，避免重复建模
- 先补迁移脚本，再改实体、Mapper 和 service
- 涉及历史数据时，要同步考虑默认值和回填策略
- 变更完成后补一条简短的数据库变更说明，方便回滚和排障

## 7. 配置与环境规范

### 7.1 后端配置

当前后端配置已按环境拆分：

- `application.properties`：默认配置
- `application-local.properties`：本地开发
- `application-demo.properties`：演示环境
- `application-prod.properties`：生产环境

后续开发应遵循以下原则：

- 数据库地址、账号、密码、上传路径、文件大小等敏感项都通过环境变量覆盖
- 不要把真实环境参数写死到代码里
- 文件上传、模板文件路径等必须保持可配置

### 7.2 前端配置

- 前端命令以 `npm run dev / build / preview / test` 为准
- 新增环境变量时要同步说明用途
- 接口地址、认证相关配置应避免硬编码

## 8. 新功能开发流程

新增业务建议按下面顺序推进：

1. 明确业务边界和数据流
2. 确认是否需要新增表、字段或索引
3. 定义 `DTO / Query / VO`
4. 实现 `Controller -> Service -> Mapper` 链路
5. 补充全局异常、参数校验和权限判断
6. 前端同步补路由、页面和接口调用
7. 补测试用例
8. 更新文档

### 8.1 后端实现清单

- 新增请求对象
- 新增返回对象
- 新增实体或复用已有实体
- 新增 service 接口与实现
- 新增 mapper 与 SQL 映射
- 新增 controller 接口
- 新增或更新数据库迁移脚本
- 新增接口测试

### 8.2 前端实现清单

- 新增路由
- 新增页面视图
- 新增表单校验与状态展示
- 新增接口调用封装
- 补充路由守卫或权限控制
- 补充前端逻辑测试

## 9. 测试规范

### 9.1 后端测试

当前后端测试主要覆盖：

- 认证接口
- 管理员接口
- 用户端业务接口
- 通知公告接口
- token 并发与失效逻辑

后续新增接口时，至少补下面三类测试：

- 正常路径
- 参数错误或边界条件
- 未登录或无权限场景

### 9.2 前端测试

当前前端已有路由守卫测试，后续建议继续补：

- 核心表单校验
- 路由跳转
- 管理端关键交互逻辑

### 9.3 回归要求

- 每次改动完成后至少执行相关单测
- 影响多模块时要补端到端回归检查
- 涉及数据库变更时要确认初始化、兼容和回滚路径

## 10. 交付与提交流程

建议采用下面的最小提交流程：

1. 本地完成实现
2. 运行后端测试
3. 运行前端测试
4. 进行必要的手工验证
5. 更新相关文档
6. 再提交代码

### 10.1 提交前检查

- 是否存在未处理的校验缺失
- 是否存在接口返回结构不一致
- 是否存在数据库脚本缺失
- 是否存在前端路由和后端路径不一致
- 是否存在测试遗漏

## 11. 目前建议优先补强的点

从仓库现状看，后续最值得优先补强的地方是：

- 统一前后端接口层的路径常量管理，减少手写路径出错
- 给新增 DTO 补齐更完整的参数校验
- 对关键业务补充事务级测试
- 为数据库变更补更细的迁移说明
- 把 `frontend/README.md` 从模板文档升级为项目文档

## 12. 结论

这套工程约定可以概括为一句话：

> 业务改动先走模型和接口设计，再走实现和测试，最后更新数据库和文档。

按这个顺序推进，可以把当前项目从“能跑”稳定过渡到“可维护、可回归、可持续扩展”。
